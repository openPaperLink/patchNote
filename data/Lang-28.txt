The bug was caused by the method's failure to handle Unicode supplementary characters (code points > `0xFFFF`), which require two `char` values (a surrogate pair) to be written, but the original implementation only wrote a single `char`. This led to incorrect output when unescaping numeric entities for supplementary characters, resulting in invalid or incomplete character sequences that could cause downstream errors or incorrect text rendering. Evidence of the issue is demonstrated by the `testSupplementaryUnescaping` unit test, where the input `&#68642;` (code point `0x10C22`) failed to produce the expected surrogate pair `\uD803\uDC22`. The bug was resolved by adding a condition to check if the code point exceeds `0xFFFF` and, if so, using `Character.toChars(entityValue)` to write the surrogate pair, while retaining the original behavior for code points within the Basic Multilingual Plane.
